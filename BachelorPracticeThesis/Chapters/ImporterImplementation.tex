\chapter{Realizace projektu Importér}
\label{chap:importer_implementation}

Tato kapitola popisuje technickou realizaci klíčové části odborné praxe – modulu Importér. Řešení je rozděleno do tří vrstev: databázové, aplikační (backend) a prezentační (frontend).

\section{Analýza a návrh databáze}

Databázová vrstva je implementována na platformě Microsoft SQL Server \cite{sqlserver_docs} pod vyhrazeným schématem \texttt{importer}. Architektura byla navržena s důrazem na konfigurovatelnost, což umožňuje definovat nové typy importů bez nutnosti zásahů do zkompilovaného kódu aplikace.

Model dat se skládá ze tří logických celků:
\begin{enumerate}
    \item \textbf{Definice struktury:} Metadata popisující, jaká data lze importovat (kontexty, sloupce).
    \item \textbf{Konfigurace šablon:} Uložení uživatelských nastavení a mapování.
    \item \textbf{Provozní data:} Logování průběhu importu a auditní stopy.
\end{enumerate}

\subsection{Klíčové entity a vztahy}

Základním stavebním kamenem je tabulka \texttt{CTypy}, která slouží jako kořenový číselník pro logickou separaci dat. Na ni navazuje hierarchická struktura definovaná v tabulce \texttt{Kontext}. Významným prvkem návrhu je rekurzivní vazba v tabulce \texttt{Kontext} přes sloupec \texttt{IdKontextNadrizeny}, což umožňuje modelovat vnořené objekty (např. Subjekt $\rightarrow$ Adresa $\rightarrow$ Kontakt).

\subsection{Systémová infrastruktura a auditování}
Všechny tabulky v rámci schématu \texttt{importer} sdílejí jednotnou sadu systémových sloupců, které zajišťují auditní stopu a integritu dat. Mezi tyto sloupce patří identifikátory autora záznamu (\texttt{SysZadal}), časové značky a sloupec \texttt{SysRowVersion} sloužící k optimistickému řízení souběhu. 


\subsection{Logické celky datového modelu}
Na základě funkčnosti lze datový model rozdělit do několika klíčových oblastí, které společně zajišťují flexibilitu celého importního modulu.

\subsubsection{Definice hierarchie a struktury}
Základem pro definici toho, co lze do systému importovat, jsou tabulky \texttt{CTypy}, \texttt{Kontext} a \texttt{KontextSloupce}.
\begin{itemize}
    \item \textbf{CTypy:} Slouží jako kořenový číselník a diskriminátor pro logickou separaci různých domén importu (např. spisy, subjekty či dokumenty).
    \item \textbf{Kontext:} Reprezentuje kategorii importovatelných dat — například Subjekt, Adresa nebo Kontakt. Díky rekurzivní vazbě \texttt{IdKontextNadrizeny} mohou být kontexty vzájemně vnořeny a tvoří tak stromovou strukturu (např. Subjekt $\rightarrow$ Adresa $\rightarrow$ Kontakt). Každý kontext zároveň slouží jako nadřazená kategorie pro svá konkrétní datová pole, která jsou evidována v tabulce \texttt{KontextSloupce}.
    \item \textbf{KontextSloupce:} Každý záznam v této tabulce představuje jedno konkrétní pole patřící do nadřazeného kontextu — například pole \textit{Jméno} nebo \textit{IČO} v kontextu Subjekt. Kromě datového typu obsahuje také identifikátor UI komponenty, což umožňuje dynamické vykreslování odpovídajících prvků (např. dropdown, datepicker) v prezentační vrstvě aplikace.
\end{itemize}

\subsubsection{Konfigurace šablon a mapování}
Procesní nastavení a propojení vstupních dat s vnitřní strukturou systému zajišťují entity \texttt{ImportSablony} a \texttt{SablonyMapovaniSloupcu}. Šablony uchovávají parametry jako režim dávkového zpracování nebo kritéria pro párování existujících subjektů. Tabulka \texttt{SablonyMapovaniSloupcu} pak funguje jako komplexní asociační entita, která propojuje tři nezávislé hierarchie: strukturu kontextů, zdrojové sloupce vstupního souboru a konkrétní výskyty kontextů v šabloně. Podporuje definici statických hodnot i pokročilé transformační logiky uložené v JSON formátu. Jejím hlavním úkolem je uložit stav šablony, tedy uchovat všechny její kontexty a sloupce, díky čemuž můžeme do databáze uložit kompletní informaci o hierarchii celé šablony.

\section{Serverová část a aplikační logika (Add-in)}

Serverová část modulu Importér je implementována v jazyce C\# jako komponenta systému EFabric. Hlavním úkolem této vrstvy je orchestrace datového toku mezi uživatelským rozhraním, interním API systému Evolio a SQL databází. Logika je rozdělena do dvou funkčních celků (Add-inů), které zajišťují oddělení přípravné a realizační fáze importu.

\subsubsection{Fáze 1: Příprava importu (PrepareImport)}
V první fázi odesílá frontend metadata o souboru a zvolenou šablonu mapování. Backend volá standardní importní API systému Evolio s parametrem \texttt{PouzeUlozit = true}. Systém v tomto režimu neprovádí zápis do ostrých agend, ale parsuje vstupní Excel/CSV soubor, provádí validace a výsledná data ukládá do dočasných tabulek logu (\texttt{Logy} a \texttt{LogyPolozky}). Výsledkem tohoto volání je unikátní identifikátor \texttt{IdLogSeq} (nebo \texttt{ImportId}), který slouží jako klíč pro následné kroky.

\subsubsection{Fáze 2: Ostrý import (DirectImport)}
Jakmile uživatel potvrdí správnost připravených dat, frontend zahájí sekvenční zpracování položek. Pro každou platnou položku je vyvolána metoda \texttt{DirectImportAsync}, která využívá dříve získaný identifikátor. Tato fáze volá stejné API rozhraní, ale tentokrát pro finální uložení do databáze.

\begin{lstlisting}[language={[Sharp]C}, caption={Implementace metody pro provedení ostrého importu}]
// Metoda zajistuje finalni zapis dat do systemu Evolio
public async Task DirectImportAsync(Guid importId)
{
    // Sestaveni URL adresy pro konkretni ImportId
    var url = string.Format("api/import/v5/DirectImport?ImportId={0}", importId);
    
    // Odeslani POST pozadavku na API. 
    // V teto fazi dojde k fyzickemu vlozeni dat do SQL tabulek systemu.
    await Context.Post(url, string.Empty, CancellationToken.None);
}
\end{lstlisting}

\subsection{Implementace databázové logiky (SQL Procedura)}

Pro zajištění konzistence a výkonu při aktualizaci stavů importu využívá backend uloženou proceduru \texttt{ImporterUpdateLogyPolozkySp}. Tato procedura atomicky aktualizuje stav řádku a v případě dokončení celé dávky provádí finalizaci logu.

\begin{lstlisting}[language=SQL, caption={Uložená procedura pro aktualizaci stavu položek}]
CREATE PROCEDURE [importer].[ImporterUpdateLogyPolozkySp]
    @SysRowid [core].[SysRowIdType], -- ID radku v tabulce LogyPolozky
    @Stav nvarchar(255),            -- S (Success) / EC (Error)
    @Popis nvarchar(255) = null     -- Detailni chybova zprava
AS
BEGIN
    SET NOCOUNT ON;
    DECLARE @IdTyp INT, @IdSablonaSeq INT, @IdLogSeq INT;

    -- 1. Aktualizace stavu konkretni polozky
    UPDATE importer.LogyPolozky
    SET Stav = @Stav,
        Popis = @Popis
    WHERE SysRowId = @SysRowid;

    -- 2. Kontrola koncovych stavu a aktualizace celkoveho logu
    IF(@Stav = 'S' OR @Stav = 'EC')
    BEGIN
        SELECT @IdTyp = IdTyp, @IdSablonaSeq = IdSablonaSeq, @IdLogSeq = IdLogSeq
        FROM importer.LogyPolozky WHERE SysRowId = @SysRowid;

        -- Overeni, zda jsou jiz vsechny polozky v davce dokoncene
        IF(NOT EXISTS(SELECT 1 FROM importer.LogyPolozky l 
                      WHERE l.IdLogSeq = @IdLogSeq AND l.Stav IN ('N','R')))
        BEGIN
            -- Finalizace celeho importniho logu
            UPDATE importer.Logy SET Stav = 'D' WHERE IdLogSeq = @IdLogSeq;
        END
    END
END
\end{lstlisting}

Využití této procedury umožňuje backendu efektivně sledovat průběh importu a okamžitě reagovat na výsledky vracené z API. Pokud API vrátí chybu pro konkrétní řádek, backend zavolá proceduru se stavem \texttt{'EC'} a popisem chyby, což se ihned promítne do uživatelského rozhraní v reálném čase.

\subsection{Moderní reaktivní vzory: Signály a RxJS}

Aplikace využívá kombinaci tradiční knihovny RxJS \cite{rxjs_docs} a nového reaktivního primitiva \textbf{Signals}, představeného v Angularu 17. Signály jsou použity pro správu synchronního stavu, který nevyžaduje komplexní transformace, což zjednodušuje kód a zvyšuje čitelnost.

V rámci \texttt{ImporterService} jsou signály využity pro sledování aktuálního kontextu:
\begin{lstlisting}[language=Java, caption={Využití signálů pro správu stavu modulu}]
// Definice signálu pro sledování vybrané šablony
public selectedSablonaId = signal<number | null>(null);

// Výpočetní signál závislý na změně ID
public isSablonaSelected = computed(() => this.selectedSablonaId() !== null);
\end{lstlisting}

Pro asynchronní operace, jako je paralelní načítání číselníků při inicializaci, je nadále využíváno RxJS. Operátor \texttt{forkJoin} zajišťuje, že se aplikace vykreslí až v momentě, kdy jsou k dispozici všechna potřebná metadata:

\begin{lstlisting}[language=Java, caption={Paralelní inicializace dat pomocí forkJoin}]
forkJoin({
  typy: this.emap.get('cTypy', { IdTyp: this.idTyp() }),
  kontexty: this.emap.get('kontext', { IdTyp: this.idTyp() }),
  sablony: this.emap.get('importSablony', { IdTyp: this.idTyp() })
}).pipe(takeUntil(this.destroy$)).subscribe(data => {
  this.state.update(s => ({ ...s, ...data }));
});
\end{lstlisting}

\subsection{Optimalizace výkonu pomocí OnPush strategie}

Z důvodu vysokých nároků na výkon při zobrazování rozsáhlých mapovacích struktur (stromy s desítkami uzlů) byla u všech komponent s vyššími nároky na výkon nasazena strategie \textbf{ChangeDetectionStrategy.OnPush}. 

Tato optimalizace instruuje Angular, aby kontroloval změny v komponentě pouze v případě, že se změní její vstupní reference (\texttt{@Input}) nebo je změna explicitně vyžádána. Tím se výrazně redukuje počet cyklů detekce změn při interakci uživatele s formulářem.

\begin{lstlisting}[language=Java, caption={Definice komponenty s OnPush strategií}]
@Component({
  selector: 'ev-importer-mapping',
  templateUrl: './importer-mapping.component.html',
  styleUrls: ['./importer-mapping.component.scss'],
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class ImporterMappingComponent { ... }
\end{lstlisting}

\subsection{Implementace komplexní mapovací logiky}

Jádrem modulu je \texttt{ImporterMappingComponent} (viz Obrázek \ref{fig:importer_mapping}), která umožňuje uživateli interaktivně sestavit strukturu importu. Na rozdíl od původního desktopového řešení se zde uplatňuje princip \textit{on-demand} budování struktury:

\begin{itemize}
    \item \textbf{Dynamické instance:} Uživatel může přidávat více instancí stejného kontextu (např. více adres k jednomu subjektu). Každá instance má vlastní stav mapování sloupců.
    \item \textbf{Interaktivní správa:} Stav sbalení a rozbalení jednotlivých sekcí je udržován v \texttt{Set<string>}, což zajišťuje rychlou manipulaci i při velkém počtu prvků.
    \item \textbf{Datová transformace:} Pomocí \textit{pure pipes} dochází k transformaci dat bez nutnosti přepočítávání při každém cyklu detekce změn.
\end{itemize}

\subsection{Real-time monitoring a UX prvky}

Proces importu probíhá sekvenčně, přičemž uživatel je o postupu informován prostřednictvím \texttt{ImportProgressDialogComponent}. Tato komponenta kombinuje několik technologií pro zajištění plynulé zpětné vazby:

\begin{itemize}
    \item \textbf{Sledování času:} Využití knihovny \textit{moment.js} v sekundovém intervalu pro zobrazení doby trvání importu.
    \item \textbf{Inkrementální DataGrid:} Výsledky jsou do tabulky (\textit{DevExtreme DataGrid}) přidávány okamžitě po dokončení požadavku na konkrétní řádek.
    \item \textbf{Vizuální indikátory:} Namísto statických spinnerů jsou v dashboardu využity \textbf{skeleton loadingy}, které predikují strukturu obsahu a snižují kognitivní zátěž uživatele při čekání na data.
\end{itemize}
